# Dev Log — 2026-04-22 Sprint

**Goal of the sprint:** move the project from "uses robust statistics" to "has empirical evidence that the production detector choice is defensible", and harden the CS side (self-observability, scalability, reproducibility) so the evaluation layer has a working stack underneath it.

**Starting point:** the repo already had a full agent/server/bot/dashboard pipeline with incident lifecycle, analytics scheduler, labelled CPU fault-injection wrapper, and exported CSV/Markdown artifacts from a previous sprint. What was missing was quantitative validation: no baseline comparison, only n=3 labelled experiments, no scalability numbers, no self-observability, no reviewer-friendly sandbox, no related-work framing.

### 1. Five-detector benchmark (`starnexus-bench`)

- `server/internal/analytics/detectors.go` — new file with a `Detector` interface plus five implementations: `FixedThresholdDetector` (Nagios-style static limits with 2-sample debounce), `PlainZScoreDetector` (non-robust rolling mean/stddev), `EWMADetector` (control-chart style), `RobustShiftDetector` (offline replay of the production median/MAD + baseline-shift scheduler), and `MahalanobisDetector` (multivariate composite with diagonal-covariance approximation).
- Each detector emits `SyntheticEvent` which converts to `db.Event`, so the existing `BuildGroundTruthEvaluation` can score all five against the same labels with identical code.
- Added `RunDetectorBenchmark` that runs a detector on every node's metric series, converts events, calls the existing evaluator, and wraps the output with 2000-iteration bootstrap 95 % CIs on detection and recovery delays. Bootstrap uses a fixed-seed LCG (with upper-32-bit extraction to work around LCG low-bit bias).
- `server/cmd/starnexus-bench/main.go` — CLI driver. Writes `benchmark.json`, `benchmark.csv`, `per_experiment.csv`, and `report.md` under `analysis-output/bench/`.
- `make bench` target added.

### 2. Expanded labelled experiment matrix

- `scripts/fault-injection-matrix.sh` — wrapper around the existing single-shot `fault-injection.sh`, runs 3 reps × `{30s, 60s, 150s, 300s}` on a target node with a 120 s cool-down between trials, ~70 min wall time.
- Ran it on `jp-lisahost`, combined with the three earlier manual trials → n = 15 labelled windows across durations 30 / 60 / 90 / 150 / 300 s.

### 4. Prometheus `/metrics` endpoint (zero deps)

- `server/internal/metrics/metrics.go` — a ~350-line registry with `Counter`, `Gauge`, and summary (sum + count) primitives. Atomic-float-via-bits for lock-free updates, RW-locked map for registration, Prometheus text exposition format writer.
- HTTP middleware wraps the mux, normalises paths (`/api/nodes/{id}/details` collapsed from `/api/nodes/tokyo-dmit/details` so label cardinality stays bounded), and records `starnexus_http_requests_total` + `starnexus_http_request_seconds` (sum/count summary).
- `Server.RefreshMetrics()` populates gauges from current DB state — node status counts, active incident counts by lifecycle state, uptime — and runs on each `/metrics` hit.
- Added `/metrics` route without auth — same gating as the rest of the server (private port 8900, SSH tunnel).

### 5. End-to-end integration test

- `server/integration_test.go` — `TestEndToEndPipeline` spins up a real server via `httptest.NewServer`, posts three synthetic reports (baseline → high-CPU → recovery), and asserts the full chain: node registered online, degraded status transition, `Node degraded` status_change event, open `node_degraded` incident, recovery back to online, incident recovered, `/metrics` exposes `starnexus_http_requests_total` and populated `starnexus_nodes_total{status="online"}`.
- Catches regressions spanning db, api, analytics, and observability layers in one pass.

### 6. Scalability benchmark (`starnexus-loadtest` + `scripts/loadtest-local.sh`)

- `server/cmd/starnexus-loadtest/main.go` — a load generator that spawns N goroutines each posting `/api/report` at a configurable interval, tracks p50/p95/p99 latency, counts transport failures vs non-200 vs success separately.
- `scripts/loadtest-local.sh` boots an isolated server on a temp DB with a minimal config, runs the loadtest at 10 / 50 / 100 / 250 / 500 agents for 15 s each, writes per-size JSON summaries under `analysis-output/loadtest/`.
- **Initial runs exposed a SQLITE_BUSY storm**: at 100+ agents the success rate dropped to ≈2 % with server logs showing `SetNodeStatus error: database is locked (5) (SQLITE_BUSY)`. Root cause: the default `sql.DB` pool opens multiple connections and `PRAGMA busy_timeout` is per-connection — only the first one honoured it.
- **Fix** (`server/internal/db/db.go`): `SetMaxOpenConns(1)` plus explicit `PRAGMA busy_timeout=5000` (5 s) plus `PRAGMA synchronous=NORMAL`. Rationale documented in a comment: SQLite serializes writes at the file level, so a single pooled connection + the WAL-mode single-writer gives the best throughput under burst load without lock contention.
- After the fix: zero errors at every tested size. 500 agents at 500 ms interval = 1000 reports/sec with p99 = 109 ms.

### 8. Multivariate Mahalanobis detector — two-pass tuning

- First pass (ComponentK=2.5, CompositeK=3.5): 100 % detection on the n=3 set, but 0.903 FP/node-hour on the full 7-day window. Too noisy.
- Tuned to ComponentK=3.0, CompositeK=4.2, floor-values raised (CPU 40, Memory 50, etc.). FP rate dropped to 0.311/hour. Detection rate on the final n = 15 set stayed at 86.7 %.
- Limitation: diagonal covariance approximation. Proper MCD-based full Mahalanobis is flagged in `docs/LIMITATIONS.md` as future work.

### 9. Weight sensitivity analysis

- `scripts/validate-reliability.py` — sweeps five weight schemes over the `node_scores` components (availability 40/30/30 default, 60/20/20, 33/33/33, 30/50/20, 30/20/50), computes Kendall-tau between rankings, reports per-node composite spread.
- On the current 3-node fleet the ranking is identical across every scheme (all Kendall-tau = +1). Documented as "weak but real robustness claim" in RESULTS and LIMITATIONS.

### 11. Post-push audit — found and fixed one real bug

- Ran `go vet ./...` across all three modules. Reported `agent/internal/probe/probe.go:41`: `address format "%s:%d" does not work with IPv6`.
- The old `fmt.Sprintf("%s:%d", host, port)` format would produce ambiguous strings for IPv6 hosts (e.g. `::1:22` — should be `[::1]:22`).
- Fixed with `net.JoinHostPort(target.Host, strconv.Itoa(port))`. One-line change, commit `cee296f`.
- Pre-existing bug, not introduced this sprint, but caught and fixed during audit.

## Non-Obvious Decisions, Recorded

- **Why not use `prometheus/client_golang`?** To keep the server a zero-dependency static binary. The text exposition format is trivial to emit; the middleware is ~30 lines; the primitives are ~200 lines. Supply-chain surface is zero, build time unaffected. Docs flag this in LIMITATIONS as "not feature-complete" (no histograms, no native summaries) but all the production-relevant cardinality is covered.
- **Why `SetMaxOpenConns(1)` instead of DSN-level pragmas?** Initially tried `file:db?_pragma=busy_timeout(5000)` — the modernc.org/sqlite driver does not honour that URI form. Falling back to `SetMaxOpenConns(1)` routes every query through a single connection whose pragmas stick, plus fully serialises writes which matches what SQLite does internally anyway. Reads don't parallelise, but read throughput isn't the bottleneck for a VPS monitoring fleet.
- **Why offline replay for all detectors, including the production one?** Apples-to-apples. The live production scheduler runs every 5 min on a moving wall clock; replay runs on the same stored time series at the same cadence. Same input, different decision logic, scores directly comparable.
- **Why report bootstrap CIs at n = 15?** Because mean-only comparisons at small n are misleading, and readers who know statistics will ask. CIs also make it immediately obvious which detectors are statistically indistinguishable (`fixed_threshold` and `mahalanobis` 95 % CIs overlap heavily on delay).
- **Why commit `docs/figures/` instead of regenerating on demand?** GitHub renders inline PNGs from tracked paths. A reviewer landing on the repo sees the benchmark table AND the figures without running anything. `analysis-output/figures/` stays gitignored for the live working copy.
- **Why add `!docs/dev-log-20260422.md` to .gitignore?** The existing `docs/dev-log-*.md` pattern was written when dev logs contained private VPS IPs and tokens. This sprint's log is scrubbed and intentionally public, so it gets a single-file exception rather than weakening the pattern.

## Open Items / Future Work

Honest list of things this sprint did NOT do:

- **Proper MCD-Mahalanobis** — the current `mahalanobis` detector is diagonal-covariance. Full covariance with Minimum Covariance Determinant trimming is the rigorous version.
- **Changepoint detection** — baseline-shift analysis still uses the rolling recent-vs-baseline median heuristic. CUSUM or Bayesian online changepoint (Adams & MacKay 2007) would be the principled upgrade.
- **Memory and network fault injection** — CPU-only for safety. Memory pressure and tc-netem latency injection need a disposable test node with safer rollback.
- **Longitudinal reliability-score validation** — split-half predictive validation needs ≥1 month of data on ≥5 nodes. Currently have ~1 week on 3 nodes.
- **Bot integration test** — the integration test covers server/api/db/metrics. Bot-side tests are still only unit-level in `bot/internal/config/`.

All of these are enumerated in `docs/LIMITATIONS.md` with reasoning for why they were out of scope for this sprint.
