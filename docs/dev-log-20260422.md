### 6. Scalability benchmark (`starnexus-loadtest` + `scripts/loadtest-local.sh`)

- `server/cmd/starnexus-loadtest/main.go` — a load generator that spawns N goroutines each posting `/api/report` at a configurable interval, tracks p50/p95/p99 latency, counts transport failures vs non-200 vs success separately.
- `scripts/loadtest-local.sh` boots an isolated server on a temp DB with a minimal config, runs the loadtest at 10 / 50 / 100 / 250 / 500 agents for 15 s each, writes per-size JSON summaries under `analysis-output/loadtest/`.
- **Initial runs exposed a SQLITE_BUSY storm**: at 100+ agents the success rate dropped to ≈2 % with server logs showing `SetNodeStatus error: database is locked (5) (SQLITE_BUSY)`. Root cause: the default `sql.DB` pool opens multiple connections and `PRAGMA busy_timeout` is per-connection — only the first one honoured it.
- **Fix** (`server/internal/db/db.go`): `SetMaxOpenConns(1)` plus explicit `PRAGMA busy_timeout=5000` (5 s) plus `PRAGMA synchronous=NORMAL`. Rationale documented in a comment: SQLite serializes writes at the file level, so a single pooled connection + the WAL-mode single-writer gives the best throughput under burst load without lock contention.
- After the fix: zero errors at every tested size. 500 agents at 500 ms interval = 1000 reports/sec with p99 = 109 ms.

### 8. Multivariate Mahalanobis detector — two-pass tuning

- First pass (ComponentK=2.5, CompositeK=3.5): 100 % detection on the n=3 set, but 0.903 FP/node-hour on the full 7-day window. Too noisy.
- Tuned to ComponentK=3.0, CompositeK=4.2, floor-values raised (CPU 40, Memory 50, etc.). FP rate dropped to 0.311/hour. Detection rate on the final n = 15 set stayed at 86.7 %.

After the previous commits, I ran an `Explore` subagent in "very thorough" mode across the backend + frontend and got 27 findings. This commit closes out all critical and high-severity items, plus the statistical-rigor and CI items, in one batch.

**Security / correctness.**

- `scripts/fault-injection.sh`: builds the label JSON via `jq --arg`, so backslashes or quotes in a node-id cannot corrupt `experiments.jsonl`. The remote SSH invocation now passes the target path and JSON as positional args to `bash -s`, not as interpolated strings — closes the shell-injection vector on `SERVER_LABELS_PATH`. `jq` is now a hard dependency of the operator's local machine.
- `server/internal/metrics/metrics.go`: `MaxLabelsPerMetric = 10 000` cap on `AddCounter`, `SetGauge`, `ObserveSummary`. - `PlainZScoreDetector.Process`: explicit warmup guard (`if idx < warmup+MinHold { return false }`) before the rolling statistics are even computed. The earlier structure relied on the edge-trigger scanner rejecting each sample individually — same end result, but fragile under misconfigured `Window` / `MinHold`. Added `minPlainZScoreWarmup = 20` constant with a comment explaining the 10-minute floor at 30 s cadence.
- `MahalanobisDetector`: expanded the comment block to document why the diagonal-covariance approximation is the right default for this sample budget. ## Non-Obvious Decisions, Recorded

- **Why not use `prometheus/client_golang`?** To keep the server a zero-dependency static binary. The text exposition format is trivial to emit; the middleware is ~30 lines; the primitives are ~200 lines. Supply-chain surface is zero, build time unaffected. Docs flag this in LIMITATIONS as "not feature-complete" (no histograms, no native summaries) but all the production-relevant cardinality is covered.
- **Why `SetMaxOpenConns(1)` instead of DSN-level pragmas?** Initially tried `file:db?_pragma=busy_timeout(5000)` — the modernc.org/sqlite driver does not honour that URI form. Falling back to `SetMaxOpenConns(1)` routes every query through a single connection whose pragmas stick, plus fully serialises writes which matches what SQLite does internally anyway. Reads don't parallelise, but read throughput isn't the bottleneck for a VPS monitoring fleet.
- **Why offline replay for all detectors, including the production one?** Apples-to-apples. The live production scheduler runs every 5 min on a moving wall clock; replay runs on the same stored time series at the same cadence. Same input, different decision logic, scores directly comparable.
- **Why paired (McNemar-flavoured) tests instead of a two-sample test for pairwise detector comparison?** The experiments are shared across detectors — each labelled window is scored by all five — so the correct null is a sign test on the discordant pairs, not an unpaired binomial. Same reasoning drives the random-sign permutation on delay deltas: we observe matched pairs, not independent samples. Ignoring the pairing would inflate variance estimates by ~2× and give softer p-values than the data actually support.
- **Why a 10 000-label cardinality cap instead of dynamic eviction?** A fixed ceiling with silent drops (and a `droppedLabels` counter) is easier to reason about than LRU/TTL eviction. It also matches operator expectations: if you see `droppedLabels > 0`, you've got an unbounded-label bug upstream, not a cache eviction issue. 10 k is well above any realistic VPS fleet × path combination.
- **Why paired tests produce "Delay p = 1.0" so often with small n?** Random-sign permutation on ≤ 5 paired observations has at most 2⁵ = 32 distinct permutations — the observed mean is always reachable by flipping signs, and the permutation distribution is very coarse. We report the value honestly rather than gating, so readers can see the small-n regime where CIs and p-values are not strongly discriminating. Scaling the experiment matrix is flagged as future work.
- **Why mirror the bench artifacts into `web/public/data/` instead of serving them dynamically?** The Pages demo has no Go backend — Cloudflare D1 alone cannot produce the bench results, which depend on the production metric history. Bundling the JSON as a static asset makes the demo self-contained and keeps the Pages deployment independent of the real server; production dashboards served by the Go server could instead point at a dynamic `/data/bench/latest` endpoint (not wired yet, future work).
- **Why report bootstrap CIs at n = 15?** Because mean-only comparisons at small n are misleading, and readers who know statistics will ask. CIs also make it immediately obvious which detectors are statistically indistinguishable (`fixed_threshold` and `mahalanobis` 95 % CIs overlap heavily on delay).
- **Why commit `docs/figures/` instead of regenerating on demand?** GitHub renders inline PNGs from tracked paths. A reviewer landing on the repo sees the benchmark table AND the figures without running anything. `analysis-output/figures/` stays gitignored for the live working copy.
- **Why add `!docs/dev-log-20260422.md` to .gitignore?** The existing `docs/dev-log-*.md` pattern was written when dev logs contained private VPS IPs and tokens. This sprint's log is scrubbed and intentionally public, so it gets a single-file exception rather than weakening the pattern.

## Open Items / Future Work

Honest list of things this sprint did NOT do, grouped by why:

**Deliberately future work — call-outs already in `docs/LIMITATIONS.md`.**

- **Proper MCD-Mahalanobis** — the current `mahalanobis` detector is diagonal-covariance. Full covariance with Minimum Covariance Determinant trimming is the rigorous version. Not done because MCD trimming is iterative and the design-trade commentary in `detectors.go` explains why the diagonal form is a defensible default for the current sample budget.
- **Changepoint detection** — baseline-shift analysis still uses the rolling recent-vs-baseline median heuristic. CUSUM or Bayesian online changepoint (Adams & MacKay 2007) would be the principled upgrade. The heuristic is cheap and defensible; the principled upgrade adds two tuning parameters and was judged a larger-than-one-sprint investment.
- **Memory and network fault injection** — CPU-only for safety. Memory pressure (`stress --vm`) and `tc-netem` latency injection need a disposable test node with safer rollback than the current `nice + timeout`. Deferred until a dedicated test VPS is online.
- **Longitudinal reliability-score validation** — split-half predictive validation needs ≥ 1 month of data on ≥ 5 nodes. Currently have ~ 1 week on 3 nodes; the `scripts/validate-reliability.py` weight-sensitivity sweep is the stopgap.

**Quality items not done this sprint.**

- **Live detector toggling in the dashboard** — the Explore audit suggested a "Detector Playground" view that re-renders live metrics under an arbitrary detector's rules. Not done: the rendering layer needs to bundle a detector implementation in JS (or add a parameterised server endpoint that runs arbitrary detectors on demand). Both are meaningful new features, out of scope for hardening.
- **Statistical internals in node detail** — surfacing per-metric robust z, baseline shift flag, and Mahalanobis composite inline in the node detail panel. The data is available via `/api/nodes/{id}/details`; no UI work was done this sprint.
- **Unified radar panels** — Fleet Radar / Experiment View / Operational Reliability currently sit as three separate panels with overlapping node lists. An information-architecture refactor into tabs would be cleaner but is UX redesign, not a bug.
- **Bot-side integration test** — the server integration test boots the real server and asserts the full `report → degraded → incident → metrics` chain. Bot has only unit-level config tests. A harness that exercises the Telegram bot against a fake Telegram API remains future work.
- **Accessibility polish on sparklines** — the SVG charts don't expose min/max/latest via `aria-label` metadata; assistive tech users get only the label "Metric sparkline". Cosmetic.
- **Architecture diagram polling footnotes** — the README ASCII diagram does not explicitly note that agent→server and bot→server are 30 s polling loops, not live streams. Cosmetic.
- **Named Docker volume for long-term data** — the sandbox `server_data:` volume is ephemeral; documented as intentional in LIMITATIONS but a `docker-compose.prod.yml` variant would be nicer.

All deliberately-deferred items are enumerated in `docs/LIMITATIONS.md` with reasoning. The quality items above are documented here and not silently hidden; `docs/RESULTS.md` already flags the "solid engineering prototype vs full statistical validation" gap.

### MCD-Mahalanobis (`server/internal/analytics/detectors_mcd.go`)

Full-covariance multivariate outlier detector with a simplified MCD (Minimum Covariance Determinant) concentration step. The existing `mahalanobis` detector uses a **diagonal** covariance approximation (`Σ_ij = 0` for `i≠j`) which throws away all cross-metric correlation. MCD-Mahalanobis fits a robust mean + full covariance on an h-subset of the rolling window using:

1. Seed the concentration with the coordinate-wise median.
2. Rank points by scaled Euclidean distance from the centre.
3. Keep the closest `h = ⌈0.75·n⌉` points.
4. Recompute mean + covariance on that h-subset; repeat 3 iterations.

Cheaper than Fast-MCD's multi-start loop and robust enough for p=4. Added a trace-proportional ridge (`cov += trace/p · 1e-3 · I`) to keep the covariance invertible when two metrics are highly correlated (real scenario: bandwidth ↔ connections co-moving during a traffic spike).

Matrix inversion is Gauss-Jordan with partial pivoting on a 4×4, written inline. `TestInvertMatrixRoundTrip` and `TestInvertMatrixSingular` verify the inverter on a tridiagonal and a rank-deficient matrix.

### CUSUM (`server/internal/analytics/detectors_cusum.go`)

Classical Page (1954) CUSUM, one-sided upper, per-metric with an OR combiner. Uses the unknown-parameter variant: standardise each sample against a rolling robust baseline (`median` + `1.4826·MAD`) on a 100-sample window, accumulate `S_t = max(0, S_{t-1} + z - K)` with K=0.5 and fire when S exceeds H=5. ARL₀≈930 samples under the null — roughly one false fire per 7.7 h per metric at 30 s cadence, before the OR combiner.

Design choices:

- **Untuned K and H**: textbook defaults, deliberately not fit to the labelled set — avoids the trained-on-test artifact.
- **Soft reset after firing**: `S_t` zeroes so the detector can re-arm on subsequent shifts rather than staying latched.
- **MinHold=2 gate**: matches the rest of the catalogue so the comparison is fair across detectors.

### Silent SSH label-push bug, discovered en route

While writing the memory script I discovered the `fault-injection.sh` server-label push had never actually worked. The prior pattern:

```bash
ssh "$SERVER_SSH" 'bash -s "$1" "$2"' _ "$SERVER_LABELS_PATH" "$LABEL_JSON" <<'REMOTE_SCRIPT'
```

relied on OpenSSH forwarding positional args to the remote `sh -c`. OpenSSH does NOT do that — it concatenates remote-command args into a single string with plain spaces and exec's them via `sh -c ONE_LONG_STRING` on the far side. The remote `sh -c` has no positional params, so `"$1"` and `"$2"` inside the script both expand to empty. The CPU script had been silently failing for every run; every "server-side experiments.jsonl" line prior to this sprint had been hand-synced by some other mechanism.

Replaced with a stdin pipe: `printf '%s\n' "$LABEL_JSON" | ssh host "cat >> path"`. This carries arbitrary JSON regardless of contents. Fixed in both CPU and memory scripts. The fix is documented in the commit message because it's the kind of subtle transport bug that would bite anyone following the pattern.

## 25. Reversed Headline — Single-Failure-Mode Bias

n = 30 bench numbers reverse the prior headline. On CPU-only (n = 15), fixed-threshold and CUSUM / Mahalanobis / MCD are statistically indistinguishable (pairwise binomial p = 1.0, all 13/15). On the combined set (CPU + memory), fixed-threshold drops to **13/30 (43 %)** while the statistical trio holds **24/30 (80 %)**. Pairwise p collapses from 1.0 to **0.001** — strong evidence that the statistical detectors genuinely generalise across failure modes.

Why: fixed-threshold's memory gate is 90 %. Memory injection peaks at 60 %. **Zero** memory experiments trigger `fixed_threshold`, even though robust z = 128 at the injection moment. A static-cap detector cannot see a pressure event below saturation, no matter how statistically obvious.

This is the single most important result in the sprint: the earlier "fixed-threshold wins" claim was a CPU-only artifact. The production design's reliance on fixed-threshold for the fast path is **less complete than previously claimed**; promoting CUSUM or Mahalanobis to the live fast-path is now a real upgrade, not just a benchmark-surrogate exercise. RESULTS.md narrative section rewritten accordingly.
