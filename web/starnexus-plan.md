# StarNexus（星枢）- 分布式节点健康监控系统

## 项目概述

一个用于监控多台 VPS 节点健康状态的系统，支持链路拓扑可视化（世界地图）、实时状态监控、故障自动告警（Telegram）、长期数据分析与节点评分。

**核心理念：** 你打开一个网页，看到世界地图上标着你所有 VPS 节点，节点之间连线显示链路状态，绿色正常红色异常，昼夜交替实时变化。手机收到 Telegram 消息："🔴 东京节点离线，疑似被墙（国外可达，国内不可达）"。

### 分模块设计（重要！）

项目拆成 4 个独立模块，每个模块可以单独开发、单独运行：

| 模块 | 仓库/文件夹 | 语言 | 独立可用 |
|------|------------|------|---------|
| `starnexus-agent` | 各 VPS 上运行 | Go | ✅ 编译为单二进制文件，scp 到 VPS 直接运行，无需装环境 |
| `starnexus-server` | 中心服务器 | Go | ✅ 即使没有前端，后端也能收数据、发告警 |
| `starnexus-web` | 前端页面 | HTML/JS | ✅ 即使后端临时挂了，页面显示最后已知状态 |
| `starnexus-bot` | Telegram Bot | Go | ✅ 独立部署在 VPS 上，可查询状态、接收告警、反向心跳监控中心 |

**为什么这样拆：**
1. AI 辅助开发时，每次只处理一个模块的代码，token 少、效率高
2. 任何一个模块出问题不影响其他模块
3. 可以逐步开发，每完成一个模块就能用

## 功能清单

### 第一阶段：数据采集 + 告警（核心 MVP）

**Agent 端采集（被动监控，零额外流量）：**
- [ ] CPU 使用率（读 `/proc/stat`）
- [ ] 内存使用率（读 `/proc/meminfo`）
- [ ] 网卡实时上行/下行速率（读 `/proc/net/dev`，两次采样算差值）
- [ ] 磁盘使用率
- [ ] 系统负载（load average）
- [ ] 当前连接数（读 `/proc/net/sockstat`）
- [ ] 进程存活检测（检测 xray/singbox 等关键进程是否在跑）

**Agent 端链路检测（极低流量）：**
- [ ] Ping 延迟 + 丢包率（ping 链路中的下一跳 VPS）
- [ ] TCPing 端口可达性（比 ICMP ping 更准确）

**Agent 端健壮性：**
- [ ] 断网缓冲队列：上报失败时数据暂存内存环形队列（最多 120 条 ≈ 1 小时），网络恢复后批量补发
- [ ] 自动重连机制：中心服务器临时不可达时持续重试，不丢数据

**中心服务端：**
- [ ] 接收所有 agent 上报数据的 HTTP API（Go net/http 标准库）
- [ ] SQLite 存储（汇总数据，不存原始秒级数据）
- [ ] 节点离线检测（超过 N 秒没上报 = 离线）
- [ ] 可配置的中心服务器地址（agent 配置文件指定）
- [ ] `/ping` 健康检查接口（供 Telegram Bot 反向心跳使用）

**Telegram Bot 告警：**
- [ ] 节点离线告警
- [ ] 节点恢复在线通知
- [ ] CPU/内存超阈值告警
- [ ] 链路延迟突增告警
- [ ] 告警防抖（同一问题不重复发消息）
- [ ] 🆕 反向心跳：Bot 每 5 分钟 ping 中心服务器的 `/ping` 接口，失败则告警"中心指挥部失联"

**预计耗时：1-3 天**

### 第三阶段：高级功能

**被墙检测：**
- [ ] Agent 本地电脑同时检测各节点
- [ ] 对比"国内能否到达"和"VPS 自身能否访问外部"
- [ ] 国外通 + 国内不通 → 判定疑似被墙，告警提示

**故障自动诊断：**
- [ ] 节点离线 → 自动判断原因分类：
  - 完全无响应（VPS 宕机）
  - Ping 通但端口不通（服务挂了）
  - 国外通国内不通（被墙）
  - 能通但极慢（被限速）

**Telegram Bot 交互式操作：**
- [ ] 发送 `/status` 查看所有节点状态概览
- [ ] 发送 `/node 东京` 查看某节点详情
- [ ] 发送 `/restart 东京 xray` 远程重启服务（agent 执行）
- [ ] 发送 `/map` 返回当前拓扑状态截图

**预计耗时：3-5 天**

### 第四阶段：数据分析 + 统计（申研加分项）

**🆕 数据降采样（Downsampling）：**
- [ ] 定时任务自动执行降采样：
  - 最近 7 天：保留 30 秒原始数据
  - 7-30 天：按小时聚合（均值、最大值、标准差 σ）
  - 30 天以上：按天聚合
- [ ] 前端根据查询时间范围自动选择对应精度的数据
- [ ] 异常峰值不会被平均掉（保留 max 和 σ）

**时间序列分析：**
- [ ] 各节点 CPU/内存/带宽的趋势图（日/周/月）
- [ ] 延迟趋势分析，检测是否在逐渐恶化
- [ ] 异常检测（基于统计方法：Z-score 或 IQR）

**节点综合评分系统：**
- [ ] 基于长期运行数据给每个节点打分
- [ ] 评分维度：可用率、平均延迟、延迟波动、带宽稳定性
- [ ] 加权综合评分（权重可配置）
- [ ] 排行榜展示

**月度报告自动生成：**
- [ ] 每月自动生成运行报告
- [ ] 包含各节点可用率、故障次数、平均延迟等统计
- [ ] 发送到 Telegram 或导出 PDF

**预计耗时：2-4 天**

## 技术选型与理由

| 组件 | 技术 | 为什么选它 |
|------|------|-----------|
| Agent | Go | 编译为单二进制文件，scp 到 VPS 直接运行，零依赖，内存占用仅几 MB |
| 后端 | Go (net/http) | 标准库自带 HTTP server 不需要框架，性能好，部署同样是单二进制 |
| 数据库 | SQLite (go-sqlite3) | 单文件、零配置、够用、方便备份 |
| 前端 | HTML + JS（原生） | 地图库生态最好，基础 HTML/JS 即可，不需要 React/Vue |
| 地图 | Leaflet + 插件 | 开源免费、轻量、插件丰富（昼夜线、自定义标记等） |
| 告警 | Telegram Bot API | Bot API 简单，消息即时送达，还能做交互式命令 |

### 为什么用 Go 而不是 Python

- **部署极简**：Go 编译出单二进制文件，scp 到任何 VPS 直接 `./starnexus-agent` 就能跑，不需要装 Python 环境、pip install、虚拟环境
- **资源占用低**：Agent 内存占用几 MB vs Python 几十 MB，对小内存 VPS 友好
- **并发天生支持**：goroutine 轻松处理定时采集 + HTTP 上报 + 链路检测并行
- **申研加分**：Go 是云原生/基础设施领域的主流语言（Docker、Kubernetes、Prometheus 都是 Go 写的），用 Go 写监控系统非常"对味"

### 前端不用 Flutter 的理由

- 地图可视化在 Web 生态最成熟（Leaflet、D3、Mapbox 等）
- Flutter Web 的地图插件远不如原生 JS
- 你不需要一个 App，浏览器打开就能看最方便
- SSH -L 本地转发访问也只需要浏览器
- Flutter 留给别的项目练手

## 关键设计决策

### 中心服务器选择

**方案：可配置，默认用本地设备，也可以指定任意 VPS**

Agent 的配置文件里写中心服务器地址：
```yaml
# agent-config.yaml
server_url: "http://192.168.1.100:8900"  # 本地设备
# server_url: "http://your-vps-ip:8900"  # 或者某台 VPS
report_interval: 30  # 每 30 秒上报一次
node_name: "tokyo-1"  # 节点显示名称
latitude: 35.6762     # 地图上的位置（手动配置更准）
longitude: 139.6503
```

如果用本地设备当中心：
- 优点：不占 VPS 资源，数据在本地安全
- 缺点：关机就收不到数据了

如果用 VPS 当中心：
- 优点：24 小时运行
- 缺点：中心本身挂了需要备用告警机制

**建议：本地设备当主中心，Telegram Bot 部署在一台 VPS 上作为独立告警备份**

### 🆕 "谁来监控监控者"问题

Telegram Bot 独立部署在一台 VPS 上，承担反向心跳任务：
- 每 5 分钟请求中心服务器的 `GET /ping` 接口
- 连续 3 次失败 → 发送 Telegram 告警："🔴 中心服务器失联！"
- 中心恢复后自动发送恢复通知
- 这样即使中心服务器断电断网，你也能第一时间知道

### 数据存储策略

```
Agent 本地（内存）：
  - 正常情况：采集后立即上报，不存本地
  - 断网时：暂存到内存环形队列（最多 120 条 ≈ 1 小时）
  - 网络恢复：批量补发暂存数据，保证数据连续性

中心服务器 SQLite：
  - 保留所有上报数据（每条几百字节）
  - 一个节点一天 ≈ 2880 条 × 200 字节 ≈ 500KB
  - 10 个节点跑一年 ≈ 1.8GB（完全可接受）
  - 🆕 定时降采样：7 天内保留原始数据，7-30 天按小时聚合，30 天以上按天聚合
  - 前端查询自动选择合适精度，避免渲染百万数据点卡死浏览器

告警日志：
  - 同时存 SQLite 和发 Telegram
  - Telegram 频道天然就是一份带时间戳的运维日志
```

### 安全性

- Agent → 中心通信带 Token 验证（配置文件中设置共享密钥）
- Web 前端通过 SSH -L 本地访问，不暴露公网
- 如果需要公网访问：用 Caddy/Nginx 反代 + HTTPS，避免裸 HTTP 被运营商或扫描器干扰
- Telegram Bot 限制只响应你的 chat_id

### Agent 部署与更新

**痛点：你最烦每次 SSH 进去手动操作**

解决方案：
1. **首次部署**：Go 交叉编译出 linux-amd64 二进制文件 + 一键安装脚本
   ```bash
   curl -sSL http://your-server/install.sh | bash -s -- --token YOUR_TOKEN --server http://your-server:8900
   ```
   脚本自动下载二进制文件、写配置、创建 systemd 服务、启动

2. **后续更新**：Agent 自带自动更新功能
   - 每次上报时顺便检查中心服务器有没有新版本（对比版本号）
   - 有新版本就自动下载替换并重启自己
   - 你只需要在中心服务器上传新版本二进制文件，所有节点自动更新

3. **Go 的优势体现**：不需要 VPS 上装任何运行环境，一个文件搞定一切

### 时间统一

- 所有 agent 上报数据统一使用 UTC 时间戳
- 前端展示时转换为用户本地时间（上海时间）
- 不依赖 VPS 系统时间，用 Go 的 `time.Now().UTC()`

## 卡关应急方案

| 可能卡关的地方 | 应急方案 | 影响 |
|--------------|---------|------|
| Go 语法不熟 | AI 辅助 + Go Tour 官方教程，边做边学 | 可能稍慢但不阻塞 |
| Go 交叉编译配置 | `GOOS=linux GOARCH=amd64 go build` 一条命令搞定 | 不会卡 |
| 昼夜地图效果太难做 | 先用普通暗色底图，昼夜效果后面加 | 不影响核心功能 |
| 城市灯光图叠加效果不好 | 用纯暗色底图 + 亮色节点标记，视觉效果也不错 | 纯美观问题 |
| IP 地理定位不准 | agent 配置文件里手动填经纬度 | 更准确 |
| WebSocket 实时推送搞不定 | 前端每 10 秒轮询 API 也完全够用 | 延迟多几秒而已 |
| 被墙检测逻辑复杂 | 先只做基础在线/离线检测，被墙检测放到第三阶段 | 不影响前两个阶段 |
| SQLite 性能不够 | 10 个节点根本不会有性能问题；真不够了换 PostgreSQL | 极不可能发生 |
| Agent 自动更新机制复杂 | 先手动 scp 更新（Go 单文件所以也不麻烦） | 临时方案也能用 |
| Telegram Bot 交互命令太多 | 先只做告警推送，交互命令后面加 | 核心告警不受影响 |
| 降采样逻辑写不对 | 先不做降采样，前端限制只查最近 7 天 | 长期数据功能延后而已 |

## 申研包装话术

**项目描述（简历用）：**

> StarNexus（星枢）: 分布式网络节点健康监控系统
>
> 设计并实现了一个基于 Agent-Server 架构的分布式节点监控平台。Agent 端采用 Go 语言开发，编译为零依赖的单二进制文件实现一键部署，通过被动采集系统指标与链路探测实现低开销数据收集，内置断网缓冲队列保证数据连续性；服务端基于 Go 标准库实现数据聚合、时间序列降采样、异常检测与自动告警；前端基于 Leaflet 实现全球节点拓扑的实时地图可视化，支持昼夜动态渲染与链路状态映射。系统集成基于统计方法的异常检测模块与多维加权节点评分模型，通过 Telegram Bot 实现告警推送与交互式运维。

**面试可以聊的技术点：**
- 为什么选 Go？（单二进制部署、低内存占用、goroutine 并发、云原生生态主流语言）
- 为什么选择 Agent 主动推送而不是中心拉取？（NAT 环境、防火墙友好）
- 断网缓冲队列怎么设计的？（环形队列、批量补发、数据不丢失）
- "谁来监控监控者"？（Telegram Bot 反向心跳机制）
- 数据降采样策略？（多精度时间窗聚合，保留 max 和 σ 不丢异常峰值）
- 数据上报频率怎么定的？（30 秒平衡实时性和资源消耗）
- 告警防抖怎么做的？（避免节点抖动导致消息轰炸）
- 被墙检测的原理？（多视角探测对比）
- 节点评分的权重怎么设计的？（可以扯统计学方法）
- 为什么用 SQLite 不用 MySQL？（单文件、零运维、足够用）
